# Feishu Proposal — 客户方案撰写技能

基于飞书 CLI，将客户沟通会议的智能纪要和文字记录，自动转化为结构化的客户方案文档并写到飞书。

## 依赖

- [lark-cli](https://github.com/larksuite/cli)（`npm install -g @larksuite/cli`）
- 已完成 `lark-cli auth login` 认证
- AI Agent（如 Claude Code、Cursor 等），用于理解会议内容并生成方案

## 触发场景

- 用户给出一份客户沟通会议的飞书文字记录 + 智能纪要（两个 docx token 或链接）
- 用户说"帮我写一版方案"、"整理成项目方案"、"客户方案"
- 用户需要把模糊的会议内容转成可直接发给客户的正式文档

## 核心工作流

### Step 1：读取会议素材

用 lark-cli 拉取飞书文档（文字记录 + 智能纪要）。

```bash
lark-cli docs +fetch --doc <transcript_doc_token> --format pretty
lark-cli docs +fetch --doc <summary_doc_token> --format pretty
```

也可以通过 `vc +notes` 命令，用会议 ID 一次获取纪要和文字记录的 doc token：

```bash
lark-cli vc +notes --meeting-ids <meeting_id>
```

token 提取：用户可能直接给链接，从 `https://xxx.feishu.cn/docx/XXXXXXXXX` 中抽出 `docx/` 后面的那段即可。

### Step 2：提炼会议要点

读完素材后，严格对齐原话提取下列字段：

- 公司基本信息：企业性质、业务范围、规模（成立时间、门店数、SKU 数、团队结构、销售额）
- 现有系统：用了什么、保留 / 替换 / 衔接
- 痛点清单（至少 5 条）
- 功能诉求清单
- 业务流程：拆分成若干条平行线路，每条线路列出顺序步骤
- 客户承诺：工期、交付时间、特殊约束
- 提到的具体数字（数量、比例、金额）必须保留

禁止脑补。如果会议没说的，列到待确认事项章节，不要臆测。

### Step 3：查飞书官方费用（实时）

每次生成方案前执行 WebSearch，查最新飞书价格：
- 飞书商业标准版 / 专业版 / 旗舰版单价
- 多维表格单表行数上限、月度自动化次数
- 推荐版本的理由

加免责声明："以上价格为公开参考资料整理，最终以飞书商务报价为准。"

### Step 4：按标准模板撰写方案

方案文件写到 `/tmp/proposal.md`，使用 `assets/template.md` 作为骨架。严格遵守以下规则：

1. 不要一级标题（`# xxx`），飞书文档 title 已是标题，一级标题会重复显示
2. 顶部抬头用四个独立段落，不用引用块：
   ```
   致：<客户单位>

   编制：<你的公司名>

   版本：v1.0

   日期：<年月>
   ```
3. 对外商务口吻：
   - "客户" → "贵司"
   - "我们" → "我司"
   - 去掉"会议纪要"、"参会人"、"草案"等内部元信息
4. 流程图用 mermaid 代码块，lark-cli 会自动转成飞书画板。业务流程用 `flowchart TD/LR`，表间关系用 `erDiagram`
5. 所有表格用 markdown table 语法，飞书会自动渲染
6. 字段级设计开头加免责声明：
   > 以下字段设计为初版建议，实施阶段会根据贵司实际业务细节灵活调整，不作为最终落地版本。

### Step 5：写到飞书

新建文档：
```bash
lark-cli docs +create \
  --title "<项目名> — 需求与解决方案 v1.0" \
  --markdown @/tmp/proposal.md
```

覆盖已有文档：
```bash
lark-cli docs +update \
  --doc <output_doc_token> \
  --mode overwrite \
  --markdown @/tmp/proposal.md
```

### Step 6：返回链接给用户

交付文档 URL，简述做了哪些事（章节数、表格数、工期）。

## 标准方案结构

详见 `assets/template.md`。结构如下：

| 章节 | 内容 | 关键要点 |
|---|---|---|
| 一、项目背景 | 贵司概况 + 现有系统 | 数字严格对齐原话 |
| 二、核心痛点与需求 | 痛点清单（≥5 条）+ 功能诉求清单 | 每条痛点一句话 |
| 三、业务流程梳理 | 多条平行线路的流程 | mermaid flowchart 画板 |
| 四、解决方案总览 | 各模块的方案说明 | 按业务模块分小节 |
| 五、合作模式 | 合作方式说明 | 搭建期 / 持续期 / 沟通 |
| 六、字段级数据模型 | 所有业务表的字段清单 | 开头加免责声明，表间关系用 mermaid erDiagram |
| 七、待确认事项 | 所有会议没拍板的事项 | 编号列表 |

章节数量可根据实际情况增减。如果客户需要报价章节，自行补充。

## 飞书版本推荐逻辑

查好价格后，按客户数据规模推荐：

- 数据量小且自动化需求少 → 基础版（免费）够用
- 需要外部协作 / 行列权限 / 较多自动化 → 商业专业版（推荐）
- 数据量大或需要大量自动化 → 商业旗舰版

## 陷阱与规避

### 文件操作陷阱

1. `$(cat file.md)` 在 file 被意外清空时会生成空文档，写文件后必须检查行数
2. Python 单行 `open('w').write(open().read().replace())` 会先截断文件，必须分两步

### 飞书渲染陷阱

3. 引用块（`> xxx`）在移动端会把多行挤成一行，顶部抬头信息用独立段落
4. 纯文本代码框渲染效果差，流程图用 mermaid 代码块（lark-cli 自动转画板）
5. 一级标题 `# xxx` 会与文档 title 重复，永远从 `## 二级标题` 开始
6. mermaid 代码块会被 lark-cli 自动转成飞书画板，业务流程用 `flowchart`，表间关系用 `erDiagram`，思维导图用 `mindmap`

### 业务逻辑陷阱

7. 不要把客户原话里没说的数字编造出来，宁可列到待确认事项
8. 飞书官方价格必须实时查 WebSearch，不要引用旧数据
9. 重写全文前务必先备份：`lark-cli docs +fetch --doc <token> > /tmp/backup.txt`
